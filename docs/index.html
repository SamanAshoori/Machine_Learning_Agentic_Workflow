Agentic ML Pipeline

Dataset-agnostic ML classification with a desktop GUI

Upload any CSV, pick a target column, and step through five stages of an ML classification pipeline. Every stage pauses for your review before proceeding — no black boxes.

📄
Any CSV
Works on any binary classification dataset. Not tied to a specific schema.
👀
Human-in-the-loop
Every stage shows results and waits for your confirmation before moving on.
💻
Desktop app
Tauri v2 shell — runs as a native window on macOS and Windows.
📈
Probability scoring
Final stage outputs per-row probabilities and Low/Medium/High/Very High segments.

1 ETL Stage

The entry point. Upload a CSV, choose the column you want to predict, then review every column and decide what to keep.

What you do

  1. Upload any .csv file.
  2. Select the target column (binary: 0/1 or two distinct values).
  3. The backend runs statistical tests (Mann-Whitney U for numerics, chi-squared for categoricals) comparing each column against the target.
  4. Review each column — set to Keep, Drop, or leave blank with an optional note.
  5. Confirm to produce the cleaned dataset.

What happens in the backend

  • Dropped columns are removed.
  • Non-numeric columns are encoded with LabelEncoder.
  • Class weights are computed (sklearn.utils.class_weight.compute_class_weight) and saved.

Outputs

data/cleaned_fraud.csv — cleaned dataset with only kept columns
data/class_weights.json — per-class weights for imbalanced datasets

2 Stats Stage

Automated feature selection using multiple statistical methods. You review the results and choose which features to pass to model training.

Methods run

MethodPurposeThreshold
Pearson correlationLinear relationship with target|r| > 0.05
Chi-squaredCategorical independence testp < 0.05
VIF (Variance Inflation Factor)Detect multicollinearityVIF < 10
Mutual InformationNon-linear dependency with targetMI > 0

Auto-selection logic

A feature is auto-selected if it passes at least one test and does not have a VIF above the threshold. You can override any selection before confirming.

Outputs

data/selected_features.json — list of feature names to use in model training

3 Model Stage

Train a Random Forest classifier with configurable hyperparameters. Review metrics before confirming.

Hyperparameters

ParameterDefaultDescription
n_estimators100Number of trees in the forest
max_depth10Maximum depth of each tree (null = unlimited)
class_weightbalancedbalanced or none
test_split0.2Fraction of data held out for testing

Metrics shown

  • ROC-AUC (test set)
  • Precision, Recall, F1 per class
  • Confusion matrix
  • Feature importances (top 20)

Outputs

data/model.pkl — serialised RandomForestClassifier (joblib)
data/model_metrics.json — full metrics dict

4 Evaluate Stage

Tune the classification threshold on a validation split. Trade off precision vs recall based on your business priority.

What it does

Sweeps classification thresholds from 0.1 to 0.9 and computes precision, recall, F1, and false-positive count at each point. Results are shown as an interactive chart.

Confirm options

FieldOptionsEffect
threshold0.0 – 1.0Probability cutoff used in scoring stage
primary_metricrecall / precision / f1Highlighted metric in the UI

Outputs

data/eval_report.json — threshold sweep results + confirmed threshold

API Reference

The FastAPI backend runs on http://localhost:8000. Interactive docs at /docs.

Session

GET/status

Returns current session state: stage statuses, current stage, confirmed outputs.

POST/reset

Clears all session state. Does not delete files in data/.

ETL

POST/upload

Upload a CSV file (multipart/form-data). Returns row/column counts and basic column info.

// Response
{
  "filename": "fraudTrain.csv",
  "rows": 1296675,
  "columns": 23,
  "column_info": [...]
}
GET/columns

Detailed per-column stats: dtype, null count, unique count, mean/std/min/max for numerics, top values for categoricals.

POST/analyze/target

Run statistical tests for all columns against the chosen target.

// Body
{ "target": "is_fraud" }
POST/confirm/etl

Apply column keep/drop decisions. Produces cleaned CSV and class weights.

// Body
{
  "target": "is_fraud",
  "columns": [
    { "column": "amt", "decision": "keep", "note": "" },
    { "column": "zip", "decision": "drop", "note": "too many uniques" }
  ]
}

Stats

POST/run/stats

Run correlation, VIF, mutual information. Returns per-feature scores and auto-selected feature list. ETL must be confirmed first.

POST/confirm/stats

Save the final feature selection.

// Body
{ "selected_features": ["amt", "city_pop", "unix_time", ...] }

Model

POST/run/model

Train the RandomForest classifier. Stats must be confirmed first.

// Body
{
  "model_type": "RandomForest",
  "hyperparameters": {
    "n_estimators": 100,
    "max_depth": 10,
    "class_weight": "balanced",
    "test_split": 0.2
  }
}
POST/confirm/model

Confirm the trained model. Same body shape as /run/model.

Evaluate

POST/run/evaluate

Threshold sweep on validation split. Model must be confirmed first. No body required.

POST/confirm/evaluate 
// Body
{ "threshold": 0.8, "primary_metric": "recall" }

Scoring

POST/run/scoring

Score the full dataset. Evaluate must be confirmed first. No body required.

GET/download/scored

Download scored_output.csv. Returns 404 if scoring has not been run.

Utility

GET/output/{stage}

Retrieve the cached output for any stage. stage is one of: etl, stats, model, evaluate, scoring.

Architecture

This app is built from three separate layers that each do a distinct job. Understanding what each one is responsible for — and what it is not responsible for — is the key to the whole thing.

The three layers explained

Layer 1 — Tauri (Rust): the window

Tauri is a framework for building desktop apps. Its job is to create a native OS window and render a web page inside it using the operating system's built-in browser engine (WebKit on macOS, WebView2 on Windows).

Tauri is not the backend. It does not process data, run ML models, or handle API requests. It is essentially a frame around a web page — the same way Electron works, but much lighter because it uses the OS browser instead of bundling Chromium.

In this project the Rust code (ui/src-tauri/src/lib.rs) does almost nothing beyond creating the window: 

// This is the entire Rust source — it just launches the window
pub fn run() {
  tauri::Builder::default()
    .setup(|app| { /* logging only */ Ok(()) })
    .run(tauri::generate_context!())
    .expect("error while running tauri application");
}

No custom Rust commands. No Rust logic. Rust's only role here is to create the native window and hand it a URL to display.

Layer 2 — Svelte + Vite: the UI

The UI is a standard web app — HTML, CSS, and JavaScript — built with Svelte and compiled by Vite. In development, Vite serves it at http://localhost:5173. Tauri points its window at that URL, so you see the Svelte app rendered inside the native window.

The UI is responsible for:

  • Displaying forms, tables, charts, and status indicators
  • Holding temporary UI state (which stage is active, what the user typed)
  • Calling the FastAPI backend when the user takes an action
  • Rendering results returned from the backend

The UI does no ML computation. It does not touch CSV files directly. Every action that changes data goes through the API.

Layer 3 — FastAPI (Python): the actual backend

FastAPI is a separate Python process you start yourself with uvicorn. It runs on http://localhost:8000. This is where all real work happens: reading CSVs, running scikit-learn, training models, scoring data.

The UI talks to FastAPI over plain HTTP — the same way a web browser talks to a server. The fact that the UI is inside a Tauri window rather than a browser tab makes no difference at all to FastAPI. It sees ordinary HTTP requests.

FastAPI is responsible for:

  • Receiving uploaded CSV files and saving them to data/
  • Running statistical tests, training models, evaluating results
  • Keeping track of pipeline progress in session state
  • Returning JSON results the UI can display

So why use Tauri at all?

Without Tauri, you would open a browser tab at localhost:5173 and use the app that way. It works — and you can still do it with npm run dev.

Tauri adds:

  • A proper window — it appears in your dock/taskbar like any other app, not a browser tab
  • Distributable installers — npm run tauri build produces a .dmg or .msi you can send to someone without them needing to install Python or Node
  • OS integration — app icons, title bar, native menus, window sizing
Note: the distributable build still requires the Python FastAPI server to be running. Tauri does not bundle Python. Packaging the Python backend into the installer is a separate problem (tools like PyInstaller can solve it, but that has not been done here yet).

Request flow

Here is exactly what happens when you click a button in the app:

1
User action in the Svelte UI
e.g. clicking "Run Stage" calls a handler in App.svelte, such as handleRunStats().
2
client.js makes a fetch() call
handleRunStats() calls runStats() from src/api/client.js, which does fetch("http://localhost:8000/run/stats", { method: "POST" }). This is a standard browser HTTP request — no Rust involved.
3
FastAPI route handler runs
api/main.py receives the request on POST /run/stats, validates that the previous stage is confirmed, then calls run_stats(target) from pipeline_runner.py.
4
Python does the computation
pipeline_runner.py reads data/cleaned_fraud.csv, runs correlation, VIF, and mutual information calculations using pandas and scikit-learn, then writes data/selected_features.json. Returns a Python dict.
5
FastAPI serialises to JSON and responds
FastAPI converts the dict to a JSON response body. The HTTP response travels back to the Svelte app.
6
Svelte stores the result and re-renders
Back in App.svelte, stageData.stats = result and stageStatus.stats = "awaiting_review" are set. Because these are Svelte 5 $state variables, the UI automatically re-renders to show the stats table and enable the Confirm button.

Annotated code path

// 1. App.svelte — user clicks "Run Stage"

  const result = await runStats();       // calls client.js
  stageData.stats = result;              // triggers UI re-render
  stageStatus.stats = "awaiting_review";
}

// 2. client.js — wraps the HTTP call

Upload walkthrough

File upload is slightly different because it sends a file, not JSON. Here is the full path:

1
User picks a file
The <input type="file"> in App.svelte fires an onchange event. handleUpload(e) reads e.target.files[0] — a browser File object.
2
client.js sends multipart form data
uploadDataset(file) creates a FormData object, appends the file, and POSTs it to /upload. No Content-Type: application/json header — the browser sets the correct multipart boundary automatically.
3
FastAPI receives the file as an UploadFile
async def upload_dataset(file: UploadFile) — FastAPI's UploadFile is a streaming wrapper. shutil.copyfileobj(file.file, f) writes the raw bytes to data/fraudTrain.csv on disk.
4
Session is reset and path is stored
session.reset() clears any previous run. session.dataset_path = str(dest) stores the file path in memory so every subsequent stage knows where to find the data.
5
Summary is returned
FastAPI calls get_dataset_summary() which reads the first 2000 rows with pandas, counts rows/columns, and returns the result as JSON. The UI stores this in datasetInfo and shows the filename + row count in the header.

State & session

Two separate state stores

The app maintains state in two places that serve different purposes:

LocationWhat it holdsLifetime
App.svelte ($state) UI state: active stage, loading flag, error message, raw result data to display Lost on page refresh / window close
api/state.py (SessionState) Pipeline progress: stage statuses, confirmed outputs, dataset path, cached stage results Lost when uvicorn restarts

The Svelte UI does not persist the API's state — it holds a copy of results after a run, but those are just for display. If you refresh the Tauri window the UI state is gone, but the FastAPI session is still intact (until you restart uvicorn or call POST /reset). The UI can recover by calling GET /status on load.

Stage gating

Both layers independently enforce that stages run in order — the backend authoritatively, the frontend as a UI convenience: 

// api/state.py — backend enforcement
def can_run_stage(self, stage):
  prev = STAGES[STAGE_ORDER[stage] - 1]
  return self.stage_status[prev] in ("confirmed", "complete")

// App.svelte — frontend enforcement (UI only)
let canRunActive = $derived(() => {
  const prev = STAGES[STAGES.indexOf(activeStage) - 1];
  return stageStatus[prev] === "confirmed" || stageStatus[prev] === "complete";
});

Even if you bypass the UI and call the API directly, the backend will reject out-of-order requests with a 400 error.

Stage status lifecycle

pending → running → awaiting_review → confirmed → complete
